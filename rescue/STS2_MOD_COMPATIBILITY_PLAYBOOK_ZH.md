# STS2 Mod 兼容性实战手册（基于本次 rescue 对话）

## 适用范围
本文档沉淀了这次排障中验证有效的兼容策略，可直接复用到其它 STS2 Mod。

前提约束：
- 不改原版代码。
- 只通过 Mod 代码 + Harmony 适配。
- 默认官方更新会改签名、改构造器、改内部调用流程。

## 本次真实踩坑

### 1）编译期 API 签名变更
现象：
- `Hook.BeforePlayPhaseStart(...)` 相关位置报 `CS7036`。

根因：
- 官方更新后方法签名变化，旧的直接调用失效。

可复用修法：
- 不要在关键流程只绑定一个签名。
- 建立兼容包装层：
  - 先尝试已知旧签名。
  - 失败后走反射回退路径（新流程）。

---

### 2）运行期“救援牌卡住”
现象：
- 救起队友后，打出的牌看起来卡住。
- 日志中重复出现 Action 异常。

关键信号：
- `MissingMethodException: Constructor on type HookPlayerChoiceContext not found`

根因：
- `HookPlayerChoiceContext` 构造器在版本间变化。
- 代码按旧构造方式实例化，异常沿 Action 链抛出，导致出牌流程异常结束。

可复用修法：
- 构造器兼容逻辑必须“容错且不抛出”。
- 按“参数类型”匹配构造参数，而不是写死参数位次。
- 枚举参数优先按名称匹配（如 `Combat`），否则取首个枚举值。
- 找不到兼容构造器时，优雅降级跳过该 Hook 分支，不要炸主流程。

---

### 3）Manifest 依赖配置导致加载失败
现象：
- 模组启动时报“依赖未加载”。

根因：
- 配置写成了 `"dependencies": [""]`（空字符串依赖）。

修复：
- 无依赖必须写：`"dependencies": []`

## 可复用兼容设计原则

### 原则 A：兼容层与业务层分离
- 版本适配写在 Compat 包装层。
- 玩法逻辑保持干净，不掺杂大量版本分支。

### 原则 B：先探测再执行
- 反射探测 method/ctor 是否存在。
- 探测成功才执行。
- 兼容探测失败不能抛到主出牌流程。

### 原则 C：构造器按类型映射
反射构造对象时按类型匹配：
- `AbstractModel`
- `CombatState`
- `Player`
- `ulong`
- `enum`
并利用默认参数，避免只靠“参数个数 + 顺序”。

### 原则 D：优雅降级优先
- 非关键 Hook 在版本不匹配时允许跳过。
- 保证 Action/出牌流程正常结束优先于“强行覆盖全部 Hook”。

### 原则 E：启动配置也属于兼容范畴
- Manifest 错误会在代码执行前直接失败。
- 每次版本迁移都要校验 manifest 字段。

## 未来更新的快速排障流程

1. 先编译
- 先清掉编译错误（签名、类型变更）。

2. 再复现
- 在游戏里复现“同一条坏路径”（例如救援出牌链路）。

3. 日志优先看异常
- 先搜：`ERROR`、`Exception`、`MissingMethodException`、`MissingFieldException`、`HarmonyException`。
- 先定位到你自己的第一层栈帧，再看引擎栈。

4. 用兼容包装修
- 不做一次性硬替换。
- 用“探测 + 回退 + 优雅跳过”。

5. 回归验证
- 重新 build。
- 只测这条关键坏路径是否恢复。

## 高价值日志模式速查

- `MissingMethodException`
  - 常见于方法/构造器签名漂移。

- `MissingFieldException` / `MissingMemberException`
  - 常见于私有成员重命名或移除。

- `HarmonyException`
  - Patch 目标方法名或签名变化。

- Loader 依赖错误
  - 多为 manifest 配置问题，不是玩法代码问题。

## 发版前兼容清单

- [ ] 关键流程无单点脆弱直调（必须有 fallback）。
- [ ] 反射构造路径不抛异常、可降级。
- [ ] manifest 依赖合法（无依赖即 `[]`）。
- [ ] 目标版本 DLL 下构建通过。
- [ ] 已复测历史坏路径。
- [ ] 日志中无该路径重复异常。

## 本次 rescue 额外结论

- 救援流程会触发回合启动相关 Hook，属于高脆弱链路。
- 在版本不一致时，保证 `PlayCardAction` 完成比“100% Hook 覆盖”更重要。
- “可跳过的 Hook 分支”比“整条出牌链路崩掉”更可接受。
